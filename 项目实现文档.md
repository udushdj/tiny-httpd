# 医疗叫号系统 — 项目实现文档

## 1. 项目概述

基于 LVGL 的嵌入式医疗叫号系统，由**ARM Linux 嵌入式终端**、**微信小程序**和**自研轻量化 HTTP 服务器 (tiny-httpd)** 三端协作完成患者队列管理、叫号展示和数据统计。

```
┌──────────────────────┐     HTTP GET (5s轮询)       ┌──────────────────────┐
│   ARM 嵌入式终端      │ ◄────────────────────────── │   tiny-httpd 服务器   │
│   (LVGL GUI)         │                             │   (自研 C 语言实现)   │
│   GEC6818 Cortex-A   │                             │   epoll + CGI 引擎    │
│   800×480 触摸屏      │                             │   单线程事件驱动       │
└──────────────────────┘                             └──────────┬───────────┘
                                                               │
                                                HTTP POST (CGI) │  GET (静态文件)
                                                               │
                                                    ┌──────────┴──────────┐
                                                    │   微信小程序          │
                                                    │   (WeChat MiniProg)  │
                                                    │   患者管理/统计/添加  │
                                                    └─────────────────────┘
```

## 4. ★ tiny-httpd 核心架构

> **设计目标**：在 ARM 嵌入式 Linux 上构建一个零外部依赖、事件驱动、支持 CGI 的轻量 HTTP/1.1 服务器，替代 Apache 为医疗叫号系统提供数据服务。

### 4.2 设计亮点

#### 亮点 1：单线程 epoll 事件驱动模型

采用 **one-loop-per-process** 架构，与 Nginx worker 进程同款设计模式。所有 I/O 操作（accept、read、write、CGI pipe）全部注册到同一个 epoll 实例，由单线程统一调度。

**与传统多线程模型对比：**

| 维度 | Apache Prefork (多进程) | Apache Worker (多线程) | **tiny-httpd (epoll)** |
|------|------------------------|------------------------|------------------------|
| 并发模型 | 每连接 fork() | 每连接 pthread | 单线程 I/O 多路复用 |
| 内存占用 | 每连接 ~5MB | 每连接 ~2MB | 全量 ~1MB |
| 上下文切换 | 进程切换 (昂贵) | 线程切换 | **零上下文切换** |
| C10K 能力 | 不可行 | 勉强 | **原生支持** |
| 适用场景 | 通用 Web | 通用 Web | **嵌入式/高并发** |

#### 亮点 2：异步非阻塞 CGI 引擎

传统 CGI 服务器 (如 Apache mod_cgi) 在 `fork()` 后阻塞 `waitpid()`，完全卡住工作进程。tiny-httpd 实现了 **全异步 CGI 管道**：

```
  fork() 之后：
  ┌─────────────────────────────────────────────────────┐
  │  父进程 (事件循环)                                    │
  │                                                      │
  │  1. close(pipe_out[1])  — 关闭写端                   │
  │  2. 写入 POST body 到 pipe_in → close(pipe_in)       │
  │  3. set_nonblock(pipe_out[0])  — 读端设为非阻塞       │
  │  4. epoll_ctl(epfd, ADD, pipe_out[0], EPOLLIN)       │
  │     ↑ 关键：管道 fd 注册到 epoll                      │
  │  5. 返回事件循环，继续处理其他连接                     │
  │                                                      │
  │  ... 事件循环继续 accept/read/write 其他客户端 ...     │
  │                                                      │
  │  6. epoll 通知 pipe_out[0] 可读                       │
  │     → handle_cgi_read() 读取 CGI 输出                 │
  │     → 构造 HTTP 响应 → 切换 STATE_WRITING             │
  │                                                      │
  │  7. 定时器触发 → reap_zombies()                       │
  │     → while(waitpid(-1, &st, WNOHANG) > 0)           │
  │       回收已终止的 CGI 子进程                          │
  └─────────────────────────────────────────────────────┘

  子进程 (CGI 脚本)：
  ┌─────────────────────────────────────────────────────┐
  │  1. dup2(pipe_out[1], STDOUT) → stdout 重定向到管道  │
  │  2. dup2(pipe_in[0], STDIN)   → stdin 来自管道       │
  │  3. chdir(g_docroot)          → 工作目录 = www/      │
  │  4. setenv(REQUEST_METHOD, QUERY_STRING, 等)         │
  │  5. execve(script_path)       → 执行 CGI 脚本         │
  └─────────────────────────────────────────────────────┘
```

**关键设计**：CGI 子进程 stdout 被重定向到 pipe，pipe 读端注册进 epoll。父进程在等待 CGI 输出期间**完全不阻塞**，可继续服务其他客户端。这是本项目区别于传统 CGI 实现的核心创新。

#### 亮点 3：HTTP/1.1 解析器状态机

从零实现了一个基于 **显式状态转移** 的 HTTP 请求解析器，不使用任何第三方 HTTP 库。

```
                    ┌─────────┐
           ┌──────►│  START   │
           │       └────┬─────┘
           │            │ 读取 method (GET/POST)
           │       ┌────▼─────┐
           │       │  METHOD  │
           │       └────┬─────┘
           │            │ 读取 URI + 空格处理
           │       ┌────▼─────┐
           │       │   URI    │
           │       └────┬─────┘
           │            │ 提取 ?query_string
           │       ┌────▼─────┐
           │       │  VERSION │
           │       └────┬─────┘
           │            │ 匹配 HTTP/1.x\r\n
           │       ┌────▼─────┐
    每行循环│       │  HEADER  │
           │       └────┬─────┘
           │            │ 提取 Content-Length
           │            │ 忽略其他 header
           │       ┌────▼─────┐
           │       │HEADER_END│
           │       └────┬─────┘
           │            │ \r\n\r\n 检测
           │       ┌────▼──────────┐
           │       │  BODY (POST)  │
           │       │ 按 CL 长度读  │
           │       └────┬──────────┘
           │            │ 读够 Content-Length 字节
           └────────────┘
                    PARSE_DONE → route_request()
```

状态机在 `recv()` 回调中驱动，每个字符推动一次状态转移。支持：
- **碎片化 TCP 到达**：状态机记住当前位置，不全量缓存即可工作
- **流水线保护**：同一连接请求完成后才接受下一个（Connection: close）
- **Content-Length 精确读取**：POST body 按指定字节数读取，不依赖 EOF

#### 亮点 4：堆分配连接池 + 防止 BSS 膨胀

**问题**：若用 `static conn_t connections[256]`（每个 conn_t 含 2×16KB 缓冲区），BSS 段高达 9.5MB。使用 `-O2` 编译时，glibc 的 `_FORTIFY_SOURCE` 在每个大数组末尾插入金丝雀 (canary)，启动时检查被误判为"缓冲区溢出"触发 `abort()`。

**解决方案**：将连接表从**静态大数组**改为**指针数组 + 动态分配**：

```c
// server.h — 指针数组，BSS 仅 256×8=2KB
extern conn_t *g_conns[MAX_CONN];

// server.c handle_accept — accept 时按需 calloc
conn_t *c = calloc(1, sizeof(conn_t));  // 堆分配 ~37KB
c->fd = fd;
g_conns[fd] = c;

// server.c close_conn — 连接关闭时释放
g_conns[c->fd] = NULL;
free(c);
```

**效果**：BSS 段从 9.5MB 降至 2KB，且只在真正 accept 连接时才分配内存。

#### 亮点 5：路径穿越防护

静态文件服务使用 `realpath()` 解析后做前缀匹配，防止 `../` 和 URL 编码的目录穿越攻击：

```c
char full_path[2048];
snprintf(full_path, sizeof(full_path), "%s%s", g_docroot, uri);

char resolved[2048];
if (realpath(full_path, resolved) == NULL) {
    // 检测路径穿越尝试
    if (strstr(uri, "..") || strcasestr(uri, "%2e%2e"))
        send_error(epfd, c, 403, "Forbidden");
    else
        send_error(epfd, c, 404, "Not Found");
    return;
}

// 即使 realpath 成功，也验证前缀
if (strncmp(resolved, g_docroot, strlen(g_docroot)) != 0) {
    send_error(epfd, c, 403, "Forbidden");
    return;
}
```

双重防护：`realpath` 失败 → 403/404；`realpath` 成功但路径逃逸 → 403。

#### 亮点 6：分级日志系统 + 自动轮转

实现了独立的三级日志系统：

| 日志文件 | 内容 | 级别 |
|----------|------|------|
| `access.log` | 每个 HTTP 请求的方法/URI/状态码 | INFO |
| `server.log` | 服务器启动/停止/配置变更 | INFO |
| `error.log` | 连接失败/CGI 异常/系统错误 | ERROR |

**自动轮转机制**：单个日志文件超过 10MB 时自动重命名：
```
access.log  → access.log.1  → access.log.2  → 删除
```

日志写入使用 `pthread_mutex_t` 保护，支持同时输出到文件和控制台（编译时可选关闭）。

### 4.3 连接生命周期状态机

```
       ┌──────┐
       │ FREE │ (未分配)
       └──┬───┘
          │ handle_accept(): calloc conn_t
    ┌─────▼──────┐
    │ STATE_READ │ ← epoll 监听 EPOLLIN
    └─────┬──────┘
          │ handle_read(): recv → http_parser
          │ PARSE_DONE → route_request()
    ┌─────▼─────────┐
    │ STATE_WRITING  │ ← epoll 监听 EPOLLOUT
    └─────┬─────────┘
          │ handle_write(): 非阻塞 write wbuf
          │ wpos == wlen → 全部发送完毕
    ┌─────▼──────┐
    │ STATE_CGI  │ ← epoll 监听 CGI pipe EPOLLIN
    └─────┬──────┘
          │ handle_cgi_read(): read pipe → 构造响应
          │ → 切换到 STATE_WRITING
          │
    任意状态 ──► CLOSED (close_conn)
    触发条件：EPOLLERR/EPOLLHUP/30s超时/CGI异常
```

### 4.4 路由分发机制

```c
void route_request(int epfd, conn_t *c) {
    // 1. 内置 API 端点
    if (strcmp(uri, "/api/status") == 0)
        → serve_api_status()  // 返回服务器状态 JSON

    // 2. CGI 路由
    if (strncmp(uri, "/cgi-bin/", 9) == 0)
        → access(script, X_OK) ? exec_cgi() : 404

    // 3. 静态文件
    if (method != GET)
        → 405 Method Not Allowed
    else
        → serve_file()  // realpath 防护 → 读取文件 → 发送
}
```

### 5.4 中文字体方案：lv_lib_freetype

**问题**：原有字体 `my_font_cn.c` 通过 `--symbols` 生成，仅包含 GB2312 的 6763 个预定义汉字。AI 对话模块 (DeepSeek) 返回的汉字若不在字符集中，LVGL 显示为空白。

**方案对比**：

| 方案 | 字体文件 | 覆盖 | 字号灵活性 | 维护成本 |
|------|---------|------|-----------|---------|
| 当前 `--symbols` | ~300KB | 6763 字 | 固定 | 每次加字重新生成 |
| `--range 0x4E00-9FFF` | ~650KB | 20,992 字 | 固定 | 加字号重新生成 |
| **lv_lib_freetype + TTF** | **TTF 3~5MB** | **全部 CJK** | **任意字号** | **一次配置** |

**最终选型**：[lv_lib_freetype](https://github.com/lvgl/lv_lib_freetype)，LVGL 官方的 FreeType 适配层。在 ARM Linux 上交叉编译 FreeType 库，运行时直接加载 `.ttf` 字体文件渲染。

**架构**：
```
lv_label_set_text("AI返回的中文")
   → LVGL 字形缓存未命中
     → Freetype 从 TTF 渲染字形 → 位图缓存
       → LVGL 绘制到屏幕
```

**关键代码**：
```c
#include "lv_lib_freetype.h"

lv_freetype_init(128);  // 最大缓存 128 个字形

lv_font_t *cn_font = lv_freetype_font_create(
    "/fonts/ali-puhui.ttf",          // 阿里巴巴普惠体 (免费商用)
    LV_FREETYPE_FONT_RENDER_MODE_BITMAP,
    24,                               // 基准字号，可动态缩放
    LV_FREETYPE_FONT_STYLE_NORMAL);

lv_obj_set_style_text_font(chat_label, cn_font, 0);
```

**部署**：编译时链接 libfreetype + lv_lib_freetype，字体文件随程序部署到开发板。

## 7. 数据流与 API 设计

### 7.1 患者数据完整生命周期

```
┌──────────────────────────────────────────────────────────────┐
│  1. 护士在小程序 [添加患者]                                    │
│     → POST /cgi-bin/api                                       │
│     → Body: [{"Patient_ID":"123","name":"张三",...}]          │
│     → tiny-httpd CGI fork → 写入 patient_queue.json           │
├──────────────────────────────────────────────────────────────┤
│  2. ARM 终端轮询 (5s 间隔)                                     │
│     → GET /patient_queue.json                                 │
│     → 返回 JSON 数组                                           │
│     → cJSON_Parse → 替换内存链表                               │
│     → LVGL 刷新：当前患者 + 队列列表 + 等待人数                 │
├──────────────────────────────────────────────────────────────┤
│  3. 护士小程序 [叫下一位]                                      │
│     → 本地 shift 队列首元素                                    │
│     → POST /cgi-bin/api → 更新 JSON                            │
│     → 下轮轮询时 ARM 终端自动更新显示                           │
├──────────────────────────────────────────────────────────────┤
│  4. 统计数据更新                                                │
│     → POST /cgi-bin/api?target=stats                          │
│     → ARM 终端 GET /stats.json → refresh_screen4_stats()      │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 API 端点映射

| HTTP 方法 | URL | 处理器 | 说明 |
|-----------|-----|--------|------|
| GET | `/patient_queue.json` | 静态文件 | 返回患者 JSON 数组 |
| GET | `/stats.json` | 静态文件 | 返回统计 JSON 对象 |
| GET | `/api/status` | 内置端点 | 服务器运行状态 |
| POST | `/cgi-bin/api` | CGI (bash) | 写入患者队列数据 |
| POST | `/cgi-bin/api?target=stats` | CGI (bash) | 写入统计数据 |
| GET | `/*` (其他) | 静态文件 | www/ 目录下文件 |

## 9. 设计决策记录

| # | 决策 | 备选方案 | 选择理由 |
|---|------|---------|---------|
| 1 | 自研 tiny-httpd 替代 Apache | Apache / Nginx / lighttpd | 零依赖、单文件部署、学习价值、ARM 友好 |
| 2 | epoll 单线程 + 异步 CGI | 多线程/proactor | 无锁竞争、无上下文切换、代码简洁 |
| 3 | 堆分配连接池 (指针数组) | 静态大数组 | 避免 BSS 9.5MB 膨胀和 Fortify 金丝雀误报 |
| 4 | HTTP/1.0 简洁协议 | HTTP/1.1 keep-alive | 嵌入式场景连接复用需求低，实现简单 |
| 5 | CGI 管道注册到 epoll | 阻塞 waitpid | 不阻塞其他连接的 I/O 处理 |
| 6 | shell CGI 处理 POST 写入 | C CGI / PHP-CGI | bash 零编译、直接可用、足够健壮 |
| 7 | 5 秒间隔轮询 | 长轮询/WebSocket | 实现简单可靠、无连接维护成本 |
| 8 | Patient_ID/age 兼容 string+number | 仅支持 number | 小程序 post 的数据为字符串类型 |
| 9 | 统一 Verdant Teal 设计系统 | 各端独立配色 | 三端视觉一致、专业感 |
| 10 | lv_lib_freetype TTF 动态渲染 | 预生成位图字体 | AI 回复汉字不可预测，TTF 全覆盖 |
